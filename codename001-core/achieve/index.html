代号 001 · 成就&统计策划案

📄 v2.0 📅 2026-07-07 🏷️ 设计文档 📊 成就统计
成就系统不是让玩家「刻意去完成某个目标」，而是让玩家在打了许多对局后回头看时发现「原来我已经打了这么多」。它是 retrospective 的成就感，不是 meta 任务列表。

一设计定位

成就和每日任务有本质区别：

每日任务成就统计
定位驱动玩家每天上线做指定行为回顾玩家自己的对局历史
刷新每日刷新累计不重置
有奖励是（钻石）否（纯展示）
有引导是（进度条、提示）否（没有进度条、「还差 XX 场」）
玩家感受"今天要做的事""原来我已经打了这么多"

二统计数据

2.1 对战统计

数据更新时机用途
总场次（total）每局结算排行榜排序、称号解锁
胜场（wins）胜利时胜率计算、称号解锁
当前连胜（streak）每局结算匹配调节、排行展示
最大连胜（maxStreak）连胜更新时称号解锁、个人记录
段位（rankIdx）升/降段时排行排序、称号解锁
星数/积分每局结算排行排序

排行榜胜率由服务端计算：Math.round((wins / total) × 10000) / 100

⚠ 已知问题：排行榜使用 total_games 和 wins（数据库顶层字段），个人页面用 game_data.total 和 game_data.wins。两者独立赋值，存在不一致风险。需统一数据源：total_games = game_data.total，wins = game_data.wins。

2.2 牌型记录

handCounts = [皇家同花顺, 同花顺, 四条, 葫芦, 同花, 顺子, 三条, 两对, 一对, 高牌]

每次小局打出最优牌型时，对应 cat（9-0）索引 +1。用于成就页展示玩家打出过什么牌型、各多少次，高牌型（皇家同花顺、同花顺、四条）有炫耀价值。

2.3 功能牌使用统计

skills[技能ID].uses = 该技能使用次数

每张功能牌独立记录使用次数。成就页展示最常用功能牌，同时帮助开发者了解使用率，辅助平衡性调整。

2.4 当前不追踪的数据

不追踪理由
每局详细出牌记录存储量太大，无明确用途
每局对手信息PvP 对手非好友，无追踪意义
总游戏时长需客户端上报，准确度不可靠

三成就页

从主页导航或头像菜单进入，以模态弹窗覆盖在当前页面之上。

┌─────────────────────────┐ │ 📊 成就 │ ├─────────────────────────┤ │ 📈 总览 │ 总场次: 127 │ 胜率: 62% │ 最大连胜: 8 │ 金币/钻石余额 ├─────────────────────────┤ │ 🃏 牌型记录（从大到小） │ 皇家同花顺: 0次 │ 同花顺: 3次 │ 四条: 12次 │ ...（共计 498 次牌型） ├─────────────────────────┤ │ ⚡ 功能牌使用 │ 5-Joker: 42次 │ 神偷: 38次 │ 命运一击: 35次 │ ... └─────────────────────────┘

设计意图：「展示，不是引导」。没有进度条、没有奖励、没有「还差 XX 场达到」——这些会变成 meta 任务。成就页只是玩家所有对局数据的忠实展示。

四现状与扩展

🔗 关联系统：成就统计为外装系统提供称号和头像框的解锁数据源（总场次、连胜、胜率、段位、收集率）。外装本身不维护独立进度数据。
