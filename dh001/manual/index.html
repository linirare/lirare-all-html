代号001 · 对战系统研发执行手册

本手册描述从"玩家点击开始对战"到"结算完毕回主页"的完整服务端-客户端交互、数据流和异常路径。研发可直接据此实现。

一、对战状态机

状态枚举值进入条件离开条件服务端行为客户端行为
空闲 IDLE 登录/结算后回主页 点击"开始对战" — 展示主页，按钮可用
校验中 VALIDATING 收到开战请求 校验通过/不通过 检查装备数量=2，库存≥1 按钮loading态
匹配中 MATCHING 校验通过 匹配成功/超时 加入匹配队列，启动计时器 显示匹配计时动画
对局创建 CREATING 匹配成功/兜底触发 创建成功/失败 生成game_id，洗牌，发牌，暂扣库存 发牌动画
回合进行 PLAYING 对局创建成功 回合结束/比赛结束 倒计时(默认25s)；收到出牌后判定 选牌+技能UI
回合结算 ROUND_SETTLE 双方出牌完毕 进入下回合/比赛结束 牌型判定→扣血→检查终局条件 展示结果动画(2s)
终局结算 MATCH_SETTLE 比赛结束 结算完成 胜退败扣→算奖励→输出22字段 结算页面
掉线 DISCONNECTED socket断开 重连成功/超时判负 15s等待→15-30s代打→30s判负 "重连中"提示
已结束 FINISHED 结算完成 回主页 清空对局上下文 展示主页

二、完整对战生命周期

Phase 1: 匹配阶段

步谁动作输入/校验输出/数据变更失败处理
1.0 客户端 发送开战请求 POST /match/start {} — 网络不通→toast提示
1.1 服务端 校验装备 equipped.length == 2 — → 返回 {error:"NOT_EQUIPPED", redirect:"cards"}
1.2 服务端 校验库存 stock[equipped[0]] >= 1 && stock[equipped[1]] >= 1 — → 返回 {error:"STOCK_EMPTY", card_ids:[...]}
1.3 服务端 状态检查 玩家不处于其他对局中 — → 返回 {error:"ALREADY_IN_MATCH", game_id:"xxx"}，客户端恢复对局
1.4 服务端 加入匹配队列 段位/MMR写入匹配池 客户端收到 {status:"matching", wait_time:0} —
1.5 服务端 分级匹配 0-5s: 同段±1小段
5-10s: ±1大段
10-15s: 全段位		 客户端收到实时 wait_time 15s无人→转人机兜底(1.6)
1.6 服务端 人机兜底 创建AI对手实例 match_origin="人机兜底", opponent_type="AI" —
1.7 服务端 匹配取消 玩家主动取消/断线 移出匹配队列，不扣库存 —

Phase 2: 对局创建

步谁动作详细逻辑数据变更失败处理
2.0 服务端 生成game_id UUID v4 game_id = uuid() —
2.1 服务端 初始化牌堆 创建52张标准牌数组(4花色×13点数)
Fisher-Yates洗牌		 deck = shuffle([A♠,2♠,...,K♣]) —
2.2 服务端 发手牌 deck.pop()×7→双方hand[]
牌堆空→自动洗新牌(兜底)		 p1.hand = [7张], p2.hand = [7张] —
2.3 服务端 发公共牌 deck.pop()×3→community[]
整场固定不变		 community = [3张] —
2.4 服务端 暂扣库存 双方 stock[card1]-1, stock[card2]-1
标记为 occupied(非最终扣除)		 stock-2(occupied) 扣库存失败(并发)→回滚对局创建
2.5 服务端 初始化比赛状态 round=1, N=3, hp=[3,3], win=[0,0], skill_used_this_round=[0,0], discard_pile=[] 状态对象创建 —
2.6 服务端 通知客户端 推送 {game_id, hand[7], community[3], opponent_info, round:1, N:3, hp:3, skill_used_this_round:0} — —

Phase 3: 回合循环（每回合重复）

步回合N谁动作详细逻辑数据变更
3.0 all — 服务端 补牌 双方 hand 补至7张：draw(7 - hand.length) hand补至7
3.1 all — 服务端 重置回合技能次数 skill_used_this_round = 0 skill_used_this_round=0
3.2 all — 服务端 开始出牌倒计时 25秒(可配)，超时触发AI代打 timer=25s
3.3 all — 服务端 处理跨回合状态 读取上回合写入的flag(同花信徒/拖延/蓄力)
执行延时效果→清除flag		 flag执行后清除

Phase 3.4: 选牌阶段（回合内详细）

子步谁动作详细逻辑限制/校验
3.4.0 客户端 展示手牌+公共牌 hand[7张]可点选，community[3张]只读 —
3.4.1 客户端 玩家点选手牌 点击=选中高亮，再点=取消 最多选N张(当前回合的N值)
3.4.2 客户端 选满N张→确认按钮亮 selected.length == N → 激活"确认出牌" 不满N张不可确认
3.4.3 服务端 超时代打 25s到期未确认→AI选N张出牌(穷举最优) 仅出牌，不使用技能

Phase 3.5: 技能使用（穿插在选牌阶段）

子步谁动作详细逻辑限制/校验
3.5.0 客户端 展示技能按钮 装备的2张牌显示为可用按钮 本场已用的牌灰掉不可点
3.5.1 客户端 点击技能 发送 POST /match/skill {card_id, target?} —
3.5.2 服务端 校验技能可用性 ① skill_used_this_round < 2
② card.used == false
③ round 在 card.round_limit 范围内
④ card.fail_condition 不满足
 任一不满足→返回 {error:"SKILL_UNAVAILABLE", reason:"xxx"}
3.5.3 服务端 执行技能 HANDLERS[effect.type](ctx, params) 详见功能牌执行字段表
3.5.4 服务端 记录状态 skill_used_this_round++
card.used = true
如需跨回合→写入round_flag 		—
3.5.5 服务端 广播技能使用 visibility=full → 对手看到牌名+效果
visibility=opaque → 对手只看到"对方使用了技能"
visibility=hidden → 对手不通知 		—

Phase 3.6: 出牌&判定

子步谁动作详细逻辑
3.6.0 服务端 确认出牌 客户端发 POST /match/play {selected:[card1..cardN]}
服务端记录：selected_cards = [...], selected_count = N
3.6.1 服务端 双方都出牌后→组合 每人：selected_cards(N张) + community(3张) = N+3 张
3.6.2 服务端 取最优5张 遍历 C(N+3, 5) 所有组合 → 计算每种组合的牌型评分 → 取最大
3.6.3 服务端 应用技能加成 如 lucky标记→若牌型=顺子(cat=4)则升为同花顺(cat=8)
3.6.4 服务端 比较评分向量 逐位比较 [cat, val1, val2, ...]
全相等→按花色：♠>♥>♦>♣
3.6.5 服务端 判定回合结果 胜方：无变化
败方：hp-1, win_count不变
平分：都不变(极罕见)

Phase 4: 终局判定&结算

步谁动作条件详细逻辑
4.0 服务端 检查比赛结束 hp==0 或 win_count==3 (5局3胜) →进入4.1结算
4.1 服务端 判定胜负 — 先到3胜者=winner
对方=loser
4.2 服务端 处理库存 胜方：stock[card1]+1, stock[card2]+1(退还暂扣)
败方：不退还(扣除暂扣)		 stock变更
4.3 服务端 计算奖励 金币：base_gold × 段位倍率
钻石：base_diamond(固定，不吃倍率)
宝箱：人机局胜+2败+1，PvP不积累
角色加成(PvE)：代号A金币加成/代号K胜利钻石加成 		gold_change, diamond_change, chest_delta
4.4 服务端 计算天梯 胜+1星(+25分大师+)、负-1星(-15分大师+)
青铜不掉星、连胜奖励、大段保护判定 		stars/score变更
4.5 服务端 计算任务/成就/红点 任务进度更新、成就handCounts+1、计算红点变化 quest/achieve/reddot
4.6 服务端 生成结算单 组装22字段 → settlement_id=uuid → 存储 settlement记录
4.7 服务端 推送结算结果 向双方推送完整结算JSON —
4.8 服务端 结算失败处理 异步重试(5分钟内×3次)→仍失败→邮件补偿 settlement retry queue

三、关键数据结构

3.1 对局上下文 (match_ctx)

字段类型示例值说明
game_idstring"a1b2c3d4"UUID
roundint1-5当前回合
Nint[3,3,3,4,5][round-1]本回合需出牌数
match_originstring"真人匹配"/"人机兜底"/"教学"对手来源
opponent_typestring"真人"/"AI"对手类型
deckarray[{suit:"♠",rank:13},...]剩余牌堆
communityarray[3张固定]公共牌
discard_pilearray[]弃牌堆(可回收)
removed_pilearray[]直接移出牌(不可回收,如破坏专家)
timerint25出牌倒计时秒数

3.2 玩家对局状态 (player_state)

字段类型示例值说明
handarray[7张]手牌，初始7张，每回合补至7
selectedarray[N张]本回合已选牌
hpint3→2→1→0生命值
win_countint0→1→2→3获胜回合数
skill_used_this_roundint0-2本回合已用技能次数(每回合重置)
equippedarray[card_id1, card_id2]本场装备功能牌(暂扣库存中)
card_used_mapmap{card_id1:false, card_id2:false}每张牌本场是否已使用
round_flagsmap{flush_pending:true,...}跨回合延时效果标记
disconnectedboolfalse是否处于掉线状态
disconnect_attimestampnull掉线开始时间

使用技能 POST /match/skill {game_id, card_id, target?} {effect, result} SKILL_UNAVAILABLE / ROUND_LIMIT / CARD_USED

五、关键校验规则速查

场景校验项条件不通过处理
开战装备数量equipped.length == 2弹引导→跳功能牌页
库存充足stock[card1]≥1 && stock[card2]≥1提示→跳商店
不在其它对局player.active_match == null恢复对局
选牌数量正确selected.length == N确认按钮灰
牌属于手牌selected ⊆ hand拒绝请求
回合匹配在PLAYING状态拒绝请求
超时timer > 0AI代打
技能回合次数skill_used_this_round < 2按钮灰
卡未使用card_used_map[id] == false按钮灰
回合限制round ∈ [min,max]按钮看不到
失败条件!card.fail_condition(ctx)提示无法使用
结算服务端权威客户端不计算奖励仅展示服务端结果
防重settlement_id唯一约束重复请求返回已有结果
库存一致性结算后库存 = 暂扣前 - consumed + returned异常→报警+补偿
✅ 实现建议：先实现状态机骨架+牌型判定引擎(Phase 0-2)，再接入技能系统(Phase 3.5)，最后加异常路径(掉线/重连/取消)。
